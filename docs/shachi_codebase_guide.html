Shachi (鯱) コードベース構造ガイド

LLMベースのエージェント・ベースト・モデリング(ABM)のためのモジュラーフレームワーク。エージェントを LLM / Tools / Memory / Config の4コアに分離し、gymスタイルの統一APIで環境と接続することで、再現性の高い社会・経済・認知シミュレーション実験を実現する。

Python ≥ 3.10 Hydra 設定管理 litellm 経由でLLM呼び出し Pydantic による型付きメッセージ asyncio 並行実行 uv 依存管理
全体像 アーキテクチャ ディレクトリ構成 コア抽象クラス 実行フロー 設定システム 実装済みタスク 読み解き方・参照ガイド 拡張方法 テスト・品質

1. 全体像 — 3層構造

Shachi のコードは大きく 3つの層 に分かれる。理解の起点は約360行の src/shachi/base.py で、ここに全ての抽象クラスが定義されている。残りのコードはすべてこの抽象の「実装」である。

コア層 src/shachi/base.py

8つの抽象クラス(Agent, Environment, Task, Observation, Message, Tool, ToolResponse, BaseMemory)を定義。フレームワークの契約(インターフェース)そのもの。

エージェント層 src/shachi/agent/

10タスク分のエージェント実装(各1ファイル、計約2,700行)。litellm でLLMを呼び、Structured Output または Function Calling で環境が要求する型に応答を整形する。

環境層 src/shachi/env/

10タスク分の環境実装(各1ディレクトリ)。元論文のコードを移植した依存コード(dep/ など)やデータセットも同梱。コード量の大半はここ。

実行・設定層 scripts/ + configs/

Hydra 設定 (configs/) とエントリポイント (scripts/main.py など)。task=xxx agent=yyy の組み合わせをCLIから差し替えるだけで実験を切り替えられる。
設計思想: エージェントと環境は互いのクラスを直接知らず、Observation(環境→エージェント)と str | BaseModel | None(エージェント→環境)だけでやり取りする。この疎結合により「同じエージェント実装を別環境で使う」「エージェントだけランダム版に差し替える」等が設定変更のみで可能になる。

4. コア抽象クラス (base.py)

全クラスが Pydantic ベース(または ABC)で、型安全なデータ交換を強制する。

クラス役割実装者が定義するもの
Messageエージェント間で交換されるメッセージ。time, src_agent_id, dst_agent_id(None=環境発信/ブロードキャスト)を持つ。環境側がサブクラス化し、タスク固有フィールドを追加
Observationエージェントが受け取る観測。messages, reward, response_type(期待する応答のPydantic型), tools を持つ。format_as_prompt_text()(必須)。マルチモーダル時は format_as_prompt_payload() も
Tool / ToolResponseエージェントが任意に呼べる関数ツールとその応答。並行実行されるため副作用なし(情報取得のみ)が推奨。環境を変える行動は step の戻り値で返す。環境側が parameters_type(Pydantic)と fun を定義
Environmentシミュレーション環境。gymスタイル。num_agents(), reset(), step(responses), done(), get_result()(いずれも必須)+ 任意の get_default_agent_configs()
Task複数の Environment を束ねる単位(例: 質問紙×ペルソナの全組み合わせ)。iterate_environments()(非同期イテレータ), aggregate_results()
Agent意思決定主体。フックは1つだけ。async step(observation) → str | BaseModel | None(+任意の update_config())
BaseMemoryメモリの抽象。add_record() / retrieve() / clear()。実装例: HistoryMemory(stockagent, econagent), CamelMemory(auction_arena, oasis)。エージェント側で実装・利用
応答の3形態: エージェントの step() は (1) 自由文字列、(2) observation.response_type で指定された Pydantic モデル(Structured Output か Function Calling で生成)、(3) None(行動しない)のいずれかを返す。この規約がエージェント⇄環境間の通信を標準化している。

5. 実行フロー (scripts/main.py)

エントリポイントはわずか67行。処理は以下の順に進む。

uv run scripts/main.py --config-name "config" task=psychobench agent=psychobench
  1. Hydra が設定を合成 — config.yaml + task/xxx.yaml + agent/yyy.yaml + CLI上書き。
  2. Task を instantiate — hydra.utils.instantiate(cfg.task) が _target_ のクラスを生成。
  3. 環境をバッチで取得 — task.iterate_environments() を batchsize(既定16)ごとに束ね、環境ごとにエージェント群を instantiate。環境が get_default_agent_configs() を返す場合は各エージェントへ update_config() で注入(ペルソナ等)。
  4. エピソード実行(並行) — 各環境で: 
    observations = await env.reset()
while not env.done():
    # 全エージェントの step を asyncio.gather で並行実行
    responses = {id: await agents[id].step(obs) for id, obs in observations.items()}
    observations = await env.step(responses)
return env.get_result()
    バッチ内の複数環境も asyncio.gather で同時進行。
  5. 集計と保存 — task.aggregate_results(results) の結果を outputs/hydra/<job>/<日時>/aggregated_results_episode_<i>.pkl に pickle 保存。

6. 設定システム (Hydra)

実験 = task × agent の組み合わせ。どちらも YAML の _target_ でクラス/ファクトリ関数を指すため、コードを変えずに CLI から差し替え可能。

configs/task/psychobench.yaml

_target_: shachi.env.psychobench
          .psychobench_env.PsychoBenchTask
questionnaire: ["BFI", "DTDD", ...]
num_tests: 10
max_questions_per_step: 30
shuffle_questions: True

configs/agent/psychobench.yaml

_target_: shachi.agent.psychobench
          .create_agents_structuredoutput
num_agents: null  # main.py が環境から補完
model: "openai/qwen3.6-35b-a3b"
model_api_base: "http://..."
temperature: 0.01
memory_cls_path: shachi.agent
                 .stockagent.HistoryMemory

CLI での上書き例: task.num_tests=5 agent.temperature=0.7 batchsize=30。ランダムベースライン(agent=psychobench_random)や vLLM 版(agent=sotopia_vllm)への切り替えも同様に1語で済む。出力先は outputs/hydra/${job.name}/${日時}/ に自動生成される。

7. 実装済み ABM タスク(10種)

各タスクは元論文の実験を Shachi の統一APIに移植したもの。レベルは複雑さの段階を表す(experimental_settings.md の分類): Level I = 単一エージェントの評価系(心理・認知テスト)、 Level II = マルチエージェントの経済・市場シミュレーション、 Level III = オープンエンドな社会シミュレーション。

名前の対応に注意: コード/設定のタスク名と元論文名が異なるものがある — LM_Caricature = 論文の CoMPosT、DigitMat(emergent_analogies_LLM) = 論文の EmergentAnalogies。指標や再現条件を調べるときは docs/experimental_settings.md(日本語版 .ja.md)を一次ソースにする。

各カードの「参照」欄: 環境=シミュレーションのルール本体、エージェント=LLM呼び出し、タスク設定/エージェント設定=CLIで変えられるパラメータ、指標=一次ソース。

Level I — 単一エージェントの評価系

PsychoBench Level I Huang et al., 2023

何を測るか: 心理測定尺度を使って LLM の人間らしい心理特性を測定。13次元(Big Five, Dark Triad, BSRI, GSE, EIS 等)を「パーソナリティ / 対人 / 動機 / 情動」の4カテゴリで評価。

仕組み: 尺度をプロンプトで提示 → エージェントがリッカート尺度(例1〜5)で全問に数値回答 → structured output でパース。質問順をシャッフルして複数シードで頑健性を確保。

主な設定: questionnaire(BFI, DTDD 等13種), num_tests=10, max_questions_per_step=30, shuffle_questions, max_parse_retries=10。

環境 env/psychobench/(psychobench_env.py, observation.py, questionnaires.json)
エージェント agent/psychobench.py(structured output / function calling / random の3版)
設定 configs/task/psychobench.yaml / configs/agent/psychobench.yaml(+ _random)
起動 task=psychobench agent=psychobench
指標 各サブスケールのスコア(原論文とのMAE)

LM_Caricature Level I CoMPosT / Cheng et al., 2023

何を測るか: LLM がペルソナを「カリカチュア(戯画化)」する度合い。

仕組み: シミュレーションを context / model / persona / topic の4次元で規定し、ペルソナ発話(Twitter・オンラインフォーラム風)を生成。individuation(デフォルトのペルソナと区別できるか)と exaggeration(特徴の誇張度)の2指標で評価。

主な設定: scenario=twitter(or onlineforum), model=openai/gpt-4o, num_agents=100, num_per_gen=1。

環境 env/lm_caricature/(lm_caricature_env.py)
エージェント agent/lm_caricature.py
設定 configs/task/lm_caricature.yaml / configs/agent/lm_caricature.yaml
起動 task=lm_caricature agent=lm_caricature task.scenario=onlineforum
指標 分布のソート済みスコア間のMAE(Bonnotte, 2013)

Cognitive Biases Level I Malberg et al., 2024

何を測るか: アンカリング・フレーミング等 30種の認知バイアスを LLM がどれだけ示すか。

仕組み: 200の意思決定シナリオ × 対照/処置条件でテストケースを生成し、応答差を比較。7段階リッカート or 11段階％の回答尺度、選択肢順を反転してポジションバイアスを補正。

主な設定: target_biases(30種), dataset_file_path=data/full_dataset.csv(別途DLが必要), randomly_flip_options=True。

環境 env/cognitive_biases/(cognitive_biases_env.py, analysis.py は UMAP 可視化)
エージェント agent/cognitive_biases_agent.py
設定 configs/task/cognitive_biases.yaml / configs/agent/cognitive_biases.yaml
起動 task=cognitive_biases agent=cognitive_biases
指標 バイアススコア(原論文とのMAE)。メモリ転移実験のターゲットにも使われる

EmotionBench Level I Huang et al., 2024

何を測るか: 実生活の状況に対する情動反応(怒り・不安・抑うつ・フラストレーション・嫉妬・罪悪感・恐怖・当惑の8情動)。

仕組み: PANAS 等の自己報告尺度で、①デフォルトの情動状態を測定 → ②状況プロンプトを提示 → ③情動スコアの変化を再測定。

主な設定: questionnaire=PANAS, emotion=ALL, select_count=5, seed=42。

環境 env/emotionBench/(emotionenv.py)
エージェント agent/emotionagent.py
設定 configs/task/emotionbench.yaml / configs/agent/emotionbench.yaml
起動 task=emotionbench agent=emotionbench
指標 情動スコア(原コードとのMAE)

DigitMat Level I EmergentAnalogies / Webb et al., 2023

何を測るか: ゼロショットの類推推論(流動性知能)。

仕組み: 原論文の4課題のうちマトリクス推論(数字行列)を対象に、自由記述での正答率を問題カテゴリ別に評価。

主な設定: data_path=all_problems_1thru5.npz, each_prob_num=20, target_prob_types=null(全カテゴリ)。

環境 env/emergent_analogies_LLM/digit_mat/(digitmat_env.py)
エージェント agent/digitmat_agent.py
設定 configs/task/digitmat.yaml / configs/agent/digitmat.yaml
起動 task=digitmat agent=digitmat
指標 カテゴリ別正答率(原論文とのMAE)

Level II — マルチエージェントの経済・市場シミュレーション

EconAgent Level II Li et al., 2024

何を測るか: 人間らしい振る舞いをともなうマクロ経済シミュレーション(GDP・インフレ等)。

仕組み: Zheng et al.(2022)の仮想経済(ai_economist を移植)。各エージェントがスキル/富を持ち労働・消費を決定 → 集合的にマクロ動態が生まれる。ルールベース環境が中央政府(税)+中央銀行(金利)として機能。

主な設定: num_agents=100, episode_length=240(月=20回の課税・金融サイクル), dialog_len=3, max_price_inflation=0.1。

環境 env/econagent/(econenv.py, dep/ に ai_economist)
エージェント agent/econagent.py
設定 configs/task/econagent.yaml / configs/agent/econagent.yaml(+ _staticbaseline)
起動 task=econagent agent=econagent task.episode_length=240 task.num_agents=100
指標 メモリ転移後の認知バイアススコア 等(探索的研究で使用)

StockAgent Level II Zhang et al., 2024

何を測るか: 動的な市場環境下の株式取引シミュレーション。

仕組み: イベント駆動。エージェントが順次「融資→売買→予測→フォーラム投稿・確認」を決定し、株価が日々変動。Stock A(化学・10年)/Stock B(テック・3年)をモデル化。投資スタイル4種、履歴メモリ(直近3件)、フォーラムAPI(ツール)を使う。

主な設定: num_agents=50, total_date=264, total_session=3, order_book=false。

環境 env/stockagent/(stockenv.py。procoder に依存)
エージェント agent/stockagent.py(HistoryMemory の実装例)
設定 configs/task/stockagent.yaml / configs/agent/stockagent.yaml
起動 task=stockagent agent=stockagent
指標 セッションレベルの価格動態(MAE)、ボラティリティ(価格変化率)

AuctionArena Level II Chen et al., 2023

何を測るか: 動的オークションでの戦略的計画・実行能力(資源配分・リスク管理・適応的推論)。

仕組み: 競り上げ式オークションで、エージェントが入札者として BDI(Belief-Desire-Intention)フレームに基づき意思決定。価値の意図的な過大評価で「勝者の呪い」を再現し、"profit-first" 戦略を割り当てる。

主な設定: num_auctions=10, item_order=random, bidders(予算20000・maximize_profit ×3), items(10品目、_true_value と estimated_value)。

環境 env/auction_arena/(auction_env.py, bidder.py, prompt_base.py)
エージェント agent/auction_arena.py(CamelMemory / EmptyMemory の例)
設定 configs/task/auction_arena.yaml / configs/agent/auction_arena.yaml(+ _random)
起動 task=auction_arena agent=auction_arena
指標 利益順位から算出した TrueSkill スコア(原メインエージェントとのMAE)

Level III — オープンエンドな社会シミュレーション

Sotopia Level III Zhou et al., 2024

何を測るか: オープンエンドなロールプレイ対話における社会的知能。7指標(SOC / SEC / FIN / REL / KNO / GOAL / BEL)で多次元評価。

仕組み: 2エージェントが200の社会シナリオで対話。Shachi 版の要点: 原実装は全対話履歴を毎回プロンプトに連結するが、Shachi ではメモリ管理をエージェント側の責務とし、環境は最新メッセージのみを渡す。要 Redis / Docker(uv run sotopia install)。

主な設定: SotopiaTask(設定はほぼ空で、内部でデータセット/シナリオを参照)。バッチ実行例では batchsize=30。

環境 env/sotopia/(environment.py)
エージェント agent/sotopia.py
設定 configs/task/sotopia.yaml / configs/agent/sotopia.yaml(+ _random / _vllm)
起動 task=sotopia agent=sotopia batchsize=30
指標 7指標(原結果とのMAE)

OASIS Level III Yang et al., 2024

何を測るか: 最大100万エージェントの SNS シミュレーション(情報伝播・集団分極化・群集効果)。

仕組み: 環境サーバ + 推薦システム + タイムエンジンで大規模SNSを再現。各ユーザーは21種のアクション空間(投稿・コメント・フォロー等)を持つ LLM エージェント。X風 / Reddit風の設定があり、本実装では X風を利用。

主な設定: config_path=gpt_example.yaml, max_steps=10, num_parallel=3。

環境 env/oasis/(snsenv.py, oasis/ に SQLite スキーマ・recsys)
エージェント agent/oasisagent.py
設定 configs/task/oasis.yaml / configs/agent/oasis.yaml
起動 task=oasis agent=oasis
指標 定性的な投稿/返信の振る舞い、メモリ転移後のバイアススコア(探索的研究)
環境ディレクトリの典型構成: *_env.py(Task と Environment の実装)+ observation.py 等の型定義 + 元論文由来のデータ・依存コード。規模はタスクにより大きく異なる(lm_caricature は小規模、cognitive_biases / oasis は多数のファイル)。

8. 読み解き方 — どれを参照してどう答えるか

質問に答えるときは、まず「フレームワーク共通の話」か「特定タスクの話」かを切り分ける。共通なら base.py と scripts/main.py、タスク固有ならセクション7のカードにある 環境 / エージェントのペアを見る。下表は質問タイプ別の参照先。

質問のタイプまず見る場所答え方のポイント
フレームワークの仕組み全般
(層構造・データの流れ)		 src/shachi/base.py → scripts/main.py 8つの抽象クラスの契約(セクション4)と gym ループ(セクション5)で説明。Observation⇄response の疎結合が肝。
「タスクXの仕組み・ルールは?」
状態がどう進むか		 env/<task>/*_env.py の reset() / step() と observation.py step で状態がどう更新され、どんな Observation(と response_type)がエージェントに渡るかを追う。
「LLM に何をどうプロンプトしているか」

9. 拡張方法 — 自分のエージェント/環境を書く

新しいエージェント

  1. shachi.Agent を継承し async step() を実装
  2. observation.format_as_prompt_payload() でプロンプト化
  3. observation.response_type があれば Structured Output(response_format=)か Function Calling(tools=)で型に合わせて返す
  4. configs/agent/ に _target_ 付き YAML を追加

新しい環境

  1. Message/Observation のサブクラスで通信の型を定義
  2. shachi.Environment を継承し gym の5フックを実装
  3. 複数条件を回すなら Task を実装(iterate_environments / aggregate_results)
  4. configs/task/ に YAML を追加

最小の実装例(じゃんけん環境と Structured Output エージェント)は README の「How to Write Your Agent / Environment」に完全なコード付きで掲載されている。
